# Этап 1. Базовый бонусный сервис

## Выбор СУБД

Рассматривались три варианта:

- PostgreSQL: надежные транзакции, зрелая блокировочная модель, индексы и аналитика, хорошая работа с денежными значениями (`numeric`), нативная поддержка Docker.
- MySQL: широко используется, но для сценариев с транзакционной проводкой и гибкой аналитикой менее удобен по сравнению с PostgreSQL.
- MongoDB: удобен для документо-ориентированных данных, но для строго транзакционной бонусной проводки и финансовой целостности требует дополнительных компромиссов.

Выбран PostgreSQL как наиболее подходящий для транзакционной бонусной системы.

## Структура данных

- `card_accounts`
  - `id` — PK
  - `card_number` — уникальный номер карты
  - `balance` — текущий баланс бонусов
  - `version` — версия записи для контроля конкурентных обновлений

- `bonus_operations`
  - `id` — PK операции
  - `card_account_id` — ссылка на карту
  - `operation_type` — `ACCRUAL`, `WRITE_OFF`, `REVERSAL`
  - `amount` — сумма операции
  - `impact_amount` — влияние на баланс (`+`/`-`)
  - `source_operation_id` — исходная операция для реверса
  - `created_at` — дата/время проводки

## Реализованные сценарии

- Начисление бонусов
- Списание бонусов
- Возврат ранее начисленных/списанных бонусов
- Получение баланса по номеру карты
- Получение истории операций по номеру карты

## REST API

Базовый путь: `/api/v1/bonuses`

- `POST /accrual`
- `POST /write-off`
- `POST /reversal`
- `GET /{cardNumber}/balance`
- `GET /{cardNumber}/operations`
